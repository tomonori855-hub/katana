## 観点

### SOLID 原則
- **S（単一責任）**: 各クラスの責務は1つに絞られているか
- **O（開放閉鎖）**: 拡張に開いて修正に閉じているか。新しい Store や Loader を追加する際に既存コードの変更が不要か
- **L（リスコフ置換）**: interface の実装を差し替えても正しく動くか。BuilderContract との関係は適切か
- **I（インターフェース分離）**: interface は小さく適切に分割されているか。不要なメソッドへの依存を強制していないか
- **D（依存性逆転）**: 上位モジュールが下位モジュールに直接依存していないか。全て抽象（interface）を通じているか

### その他の設計原則
- **デメテルの法則**: オブジェクトが直接の協力者以外の内部構造を知っていないか。メソッドチェーンで深い階層をたどっていないか
- **CQS（コマンド/クエリ分離）**: クエリ（get 等）が副作用（キャッシュ書き込み）を持っていないか。Self-Healing はクエリの中で書き込みを行うが、これは許容されるか
- **冪等性**: Queue Job が複数回実行されても安全か。並行実行で不整合は起きないか
- **ハリウッドの原則（IoC）**: 制御の反転が適切に行われているか。Loader/Store は外部から注入されているか
- **ド・モルガンの法則**: 複合条件の否定が正しく変換されているか。OR/AND の反転で論理ミスはないか
- **フェイルファスト**: 異常時に早期に失敗して明確なエラーを返すか。静かに壊れたデータを返していないか
- **DRY**: 同じロジックが複数箇所に重複していないか
- **KISS**: 不必要に複雑な設計になっていないか
- **YAGNI**: まだ必要ない機能を先回りして設計していないか
- **関心の分離**: キャッシュ管理、クエリ構築、データ読み込み、index 管理が適切に分離されているか
- **契約による設計**: interface の契約（事前条件・事後条件・不変条件）は明確か
